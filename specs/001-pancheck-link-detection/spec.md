# Feature Specification: 链接检测优化 - 接入 PanCheck 服务

**Feature Branch**: `001-pancheck-link-detection`  
**Created**: 2026-06-13  
**Status**: Draft  
**Input**: User description: "链接检测优化，当前系统对资源的有效检测不足，需要强化，强化方案，使用第三方服务来进行优化。可以查看项目 /Users/kerwin/Program/rust/tg_tool，这个项目中我使用了 panCheck 来检测资源是否有效，我希望在当前系统也配置一个 Pancheck 用于检测，而不是用当前的方案。"

## Clarifications

### Session 2026-06-13

- Q: PanCheck 接入应覆盖哪些检测点？ → A: 覆盖两处检测点（添加资源后的 scheduler 检测 + 前端详情页打开时的批量检测），统一走 PanCheck。当前前端详情页检测（`performAdvancedValidityCheck`）只支持夸克、其余平台返回 `unsupported`，属半残状态，一并通过 PanCheck 修复，并消除两套并行的检测代码路径。
- Q: 夸克网盘的检测方式如何处理？ → A: 夸克的"有效性检测"统一走 PanCheck（与其它平台一致、不消耗账号转存配额）；但 scheduler 中夸克"转存并分享"生成 `SaveURL` 的转存操作（本质是重新托管而非检测）保持不变。这意味着 scheduler 对夸克链接此前跳过有效性检测、直接转存的做法，将改为先用 PanCheck 校验原始链接有效性，避免对已失效链接做空转存。
- Q: 详情页检测结果是否写回 `Resource.is_valid`？ → A: 是，所有平台（不止夸克）在详情页检测得出的结论都持久化到 `Resource.is_valid` 并同步 Meilisearch。这取代了当前"仅夸克写回"的局部行为，使详情页巡检能即时纠正历史误判。
- Q: 两处检测点是否共享同一份缓存？ → A: 共享。两处检测点（scheduler 添加资源、前端详情页巡检）读写同一份按规范化 URL 维度的缓存，缓存有效期内互不重复请求 PanCheck，最大化命中率、保证结论一致。
- Q: 资源上如何承载四态检测结果？ → A: 实际检测只有两态（有效/失效），pending/unknown 正常不会出现、属异常瞬态。因此资源上不需要新增多态字段——`Resource.is_valid`（布尔）足以承载业务结论；仅当 PanCheck 异常不可达等导致本次未得出结论时，保持 `is_valid` 原值不动、不持久化任何"未知"状态，待稍后重试。
- Q: PanCheck 未配置/被关闭时如何处理"添加资源"流程的检测？ → A: 跳过检测、直接放行资源（不阻断主流程），并移除旧的内联 `CheckURL` 代码——PanCheck 作为唯一主检测手段，不做旧逻辑回退；可由后续巡检/手动重检补回结论。
- Q: 公开页匿名访客触发的检测如何处理？ → A: 保持现状——匿名访客打开 `/r/[key]` 详情页仍自动触发检测（接口本身无鉴权），依赖共享缓存去重 PanCheck 请求、前端 5 分钟节流控制重复触发；写回仅在链接有效↔失效翻转时发生（频次低）。不额外加身份区分或服务端限流。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 使用 PanCheck 服务检测资源链接有效性 (Priority: P1)

作为系统管理员，我希望系统的**两处**链接检测入口——① 新资源进入系统时（"待处理资源"转为正式资源的流程），② 用户在前端打开资源详情页时触发的批量检测——都统一调用已配置的 PanCheck 服务来准确判断网盘分享链接是否仍然有效，从而只保留可用的资源、过滤掉已失效（被取消、过期、违规、不存在）的链接。

**Why this priority**: 这是本次优化的核心目标。当前系统对各网盘平台的有效性检测覆盖不全、准确率不足（例如部分平台检测逻辑脆弱、依赖关键字匹配 HTML、缺少统一维护），导致失效链接混入资源库，直接影响用户检索体验和数据可信度。用集中式 PanCheck 服务替换是整个功能存在的理由，必须最先完成。

**Independent Test**: 在系统配置好 PanCheck 服务地址后，提交一个已知有效的链接和一个已知失效的链接，验证有效链接被保留、失效链接被拒绝，且判定结果与在 tg_tool 中使用 PanCheck 得到的结果一致。

**Acceptance Scenarios**:

1. **Given** PanCheck 服务地址已正确配置且服务在线，**When** 系统处理一个网盘分享链接（夸克、阿里、UC、百度、天翼、123、115、迅雷等任一受支持平台），**Then** 系统通过 PanCheck 服务判定其有效性，有效链接被保留为正式资源，失效链接被拒绝并记录原因。
2. **Given** PanCheck 服务地址已配置，**When** 某次检测因 PanCheck 异常或暂未返回明确结论而无法判定，**Then** 系统不应据此翻转资源 `is_valid`，应保持原值并允许稍后重试。
3. **Given** 某个网盘链接用当前旧方案误判为有效（实际已失效），**When** 改用 PanCheck 检测同一链接，**Then** PanCheck 能正确识别其为失效并拒绝。

---

### User Story 2 - 可配置的 PanCheck 服务接入 (Priority: P2)

作为系统管理员，我希望能通过系统配置指定 PanCheck 服务的访问地址（以及启用开关），以便我可以指向自己部署的 PanCheck 实例，并在需要时关闭该能力或回退到不检测。检测过程应支持批量提交以提升效率，并对已检测过的链接进行缓存以避免重复请求。

**Why this priority**: 没有"可配置"这一层，US1 无法落地为可运维的功能。批量与缓存则是保证大量资源场景下不拖慢系统、不冲击 PanCheck 服务的必要保障。但相对 US1 的"能不能准确检测"，这是"能不能用得好"的问题，因此排在 P2。

**Independent Test**: 在管理后台修改 PanCheck 服务地址与开关，验证系统行为随之变化（开启时走 PanCheck、关闭或未配置时按既定降级策略处理）；对同一批链接连续检测两次，验证第二次命中缓存、不再实际请求 PanCheck。

**Acceptance Scenarios**:

1. **Given** 管理员进入系统配置页面，**When** 设置 PanCheck 服务地址并启用，**Then** 后续链接检测均通过该地址调用 PanCheck 服务。
2. **Given** PanCheck 服务地址未配置或开关关闭，**When** 系统在"添加资源"流程中需要检测链接，**Then** 系统跳过检测、直接放行资源（不阻断主流程、不报错），且不回退到旧的内联检测逻辑；可由后续巡检或手动重检补回结论。
3. **Given** 一批待检测链接，**When** 系统执行检测，**Then** 链接被分组批量提交给 PanCheck，且已缓存了有效/失效结论的链接在缓存有效期内不会重复请求 PanCheck。

---

### User Story 3 - 检测结果可见与可手动复核 (Priority: P3)

作为系统管理员，我希望能在资源维度看到链接的检测结果（有效/失效）及失效原因，并能对单个资源或一批资源手动触发重新检测，以便在 PanCheck 服务升级或缓存过期后刷新结论、纠正历史误判。

**Why this priority**: 这是对前两个故事结果的可观测性与可干预能力，价值在于长期维护，但若 US1/US2 未完成则没有可展示的结果，因此优先级最低。

**Independent Test**: 对一个已有资源手动触发重新检测（忽略缓存），验证其状态与失效原因被正确更新并展示；在资源列表中能区分显示不同检测状态的资源。

**Acceptance Scenarios**:

1. **Given** 资源已完成检测，**When** 管理员查看该资源，**Then** 能看到链接的检测状态（有效/失效）以及（若有）失效原因。
2. **Given** 管理员选中一个或多个资源，**When** 点击"重新检测"，**Then** 系统忽略缓存重新调用 PanCheck 并更新这些资源的检测结论。

---

### Edge Cases

- PanCheck 服务暂时不可达（网络超时、服务重启）时，系统应如何处理当前批次？（不应将全部链接误判为失效，应保持 `is_valid` 原值不动、跳过本次更新并允许稍后重试）
- 正常情况下每条受支持平台的链接都会被 PanCheck 归入"有效"或"失效"之一；如因格式不匹配、平台识别失败等未能归类，应视为本次未得出结论（保持 `is_valid` 原值、允许重试），而非判失效。
- 同一资源包含多个链接（URL）时，若其中部分失效、部分有效，系统应如何聚合判定？（参照 tg_tool：任一失效即视为整体失效，还是按业务另有约定）
- 大量链接一次性涌入检测时，如何控制对 PanCheck 的并发与速率，避免触发限流或拖垮服务？
- 缓存中存的结论过期后，重新检测与旧结论冲突时，以最新结果为准并刷新缓存。
- PanCheck 返回的结构与预期不符（字段缺失、数组元素格式不同）时，系统应能容错解析而非崩溃。
- 由于详情页巡检会写回 `is_valid` 并同步 Meilisearch，当多个用户/会话同时打开同一资源详情页、或与 scheduler 检测并发时，需保证 `is_valid` 写入与 Meilisearch 同步的最终一致，不出现长时间不一致或重复同步抖动。
- 一次"误报失效"（PanCheck 临时抖动导致有效链接被判失效）会立即将资源标记失效并影响检索可见性，需要有可恢复手段（缓存过期后重检、管理员手动重新检测）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 能够将待检测的网盘分享链接提交给已配置的外部 PanCheck 服务，并依据其返回结果判定链接为"有效"或"失效"（二态）。
- **FR-002**: 系统 MUST 支持检测所有当前业务涉及的网盘平台（至少包括：夸克、UC、阿里云盘、百度网盘、天翼云盘、123 网盘、115 网盘、迅雷网盘），平台识别与 PanCheck 服务能力对齐。
- **FR-003**: 系统 MUST 提供可配置的 PanCheck 服务接入开关与服务地址，管理员可在后台启用/关闭并指定服务地址。
- **FR-004**: 系统 MUST 在 PanCheck 服务未配置、已关闭或不可达时按明确的降级策略处理：未配置/已关闭时跳过检测、直接放行资源（不阻断主流程、不回退旧逻辑）；不可达等异常时保持 `is_valid` 原值不动、允许稍后重试。任何情况下都不得因此中断资源处理主流程或把链接一律误判为失效。
- **FR-005**: 系统 MUST 对检测请求进行批量分组提交，并维护一份按规范化 URL 维度、带有效期的**共享**检测结果缓存，供两处检测点（scheduler 添加资源、前端详情页巡检）共同读写；已缓存"有效/失效"结论的链接在缓存有效期内任何触发点都不重复请求 PanCheck。
- **FR-006**: 系统 MUST 在调用 PanCheck 时对网络超时、非 2xx 响应等异常做容错处理：异常批次视为本次未得出结论，不翻转相关资源的 `is_valid`、保持原值，允许稍后重试，而非判失效。
- **FR-007**: 系统 MUST 在资源维度记录并展示链接的检测状态，失效时应记录失效原因。无论检测由哪个触发点发起（scheduler 添加资源、前端详情页巡检、管理员手动重新检测），得出的"有效/失效"结论都持久化到资源的 `is_valid` 字段并同步至 Meilisearch，覆盖全部受支持平台（不再限于夸克）。
- **FR-008**: 系统 MUST 支持管理员对指定资源手动触发"忽略缓存的重新检测"，并据此更新检测结论。
- **FR-009**: 系统 MUST 以本次接入替代并移除当前两套并行的检测逻辑——① `ReadyResource → Resource` 转换流程中的内联逐平台检测（`CheckURL` 及各 `checkXxx` 函数），② 前端详情页批量检测接口背后的逐平台/夸克专用检测（`performAdvancedValidityCheck` 及 `performQuarkValidityCheck`）。PanCheck 成为唯一主检测手段，不保留旧代码作为回退。
- **FR-010**: 系统 MUST 让前端详情页的批量检测接口（用户打开详情页或点击"链接检测"时触发）同样经由 PanCheck 完成检测，从而把当前仅支持夸克、其余平台一律返回"不支持"的半成品检测，扩展为覆盖全部受支持平台的有效检测。

### Key Entities *(include if feature involves data)*

- **检测状态（LinkCheckStatus）**: 表示单个网盘链接的有效性结论，**二态**：有效（valid）、失效（invalid）。pending/unknown 属异常瞬态，正常不出现，且不持久化。
- **检测结果缓存（LinkCheckResult）**: 单个已规范化链接的最近一次检测结论记录，包含链接标识、所属平台、检测状态（仅缓存有效/失效）、失效原因（若有）、检测时间、缓存过期时间。
- **资源检测状态（ResourceLinkStatus）**: 一个资源在其所含全部链接之上的聚合检测结论——任一链接失效即整体失效，全部有效即有效。承载于现有 `Resource.is_valid` 布尔字段，无需新增多态字段。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在 PanCheck 服务正确配置的前提下，对覆盖全部受支持平台的代表性链接集合（含已知有效与已知失效各若干），系统判定结果与 PanCheck 服务真实结论的一致率达到 95% 以上。
- **SC-002**: 相比当前旧方案，失效链接被误判为有效（漏检）的情况显著减少——在同一批测试链接上，误判为有效的失效链接数量下降 50% 以上。
- **SC-003**: 对同一批链接，第二次检测（命中缓存）时，实际向 PanCheck 发起的请求数量比首次下降 80% 以上。
- **SC-004**: PanCheck 服务短暂不可达（如超时、5xx）时，资源处理主流程不中断、不产生"全部失效"的误判，受影响链接 100% 保持 `is_valid` 原值不动并可被稍后重试。
- **SC-005**: 管理员能够在 2 分钟内完成 PanCheck 服务地址的配置与启停切换，且配置变更后系统行为立即生效，无需重启。

## Assumptions

- 管理员已自行部署（或将自行部署）一个可用的 PanCheck 服务实例，其对外接口与 tg_tool 项目中所使用的 PanCheck 服务兼容（提供批量链接检测能力并返回有效/失效分组）。本功能负责"接入并使用"该服务，不包含 PanCheck 服务本身的开发与部署。
- 系统调用 PanCheck 服务的鉴权方式沿用 PanCheck 服务默认的公开访问方式（与 tg_tool 一致，不额外传递 API Key/Token）；如后续 PanCheck 服务启用鉴权，可作为后续迭代补充。
- PanCheck 服务的地址、缓存有效期、批量大小、并发度等运行参数，参照 tg_tool 的成熟实践设定合理默认值（例如单批链接数、并发数、缓存有效期、单请求超时），并允许在配置中调整。
- 本次接入明确覆盖系统现有的**两处**检测触发点：① "待处理资源转正式资源"（ReadyResource → Resource）流程中的检测；② 前端打开资源详情页（或手动点击"链接检测"）时触发的批量检测接口。两处均改为统一调用 PanCheck 服务，并共享同一套检测与缓存能力。
- 现有的夸克网盘"转存并分享"特殊流程（生成 `SaveURL` 的转存，属重新托管业务）保持不变；但夸克链接的"有效性检测"（含 scheduler 转存前的前置校验、前端详情页检测）统一改由 PanCheck 完成，不再用 `Transfer()` 当作探测手段，避免消耗账号转存配额。
- `Resource.is_valid`（布尔）字段继续作为资源是否有效的最终业务标记，由新的检测路径写入；检测结论为二态（有效/失效），无需在资源上新增多态状态字段。
- 详情页检测接口（`POST /api/resources/validity/batch`）当前无鉴权、`/r/[key]` 为公开分享页，故匿名访客打开页面会触发检测。这是既有行为，本次保持不变；PanCheck 调用由共享缓存去重，`is_valid` 写回仅在结论翻转时发生，前端保留 5 分钟客户端节流，整体对 PanCheck 与 DB 的压力可控。
- 失效链接的展示与后台筛选等前端交互，沿用现有管理后台风格统一封装，不在此功能中额外扩展前端架构。
