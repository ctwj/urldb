# Feature Specification: 后台管理系统 UI/UX 整体优化

**Feature Branch**: `004-admin-ui-optimization`
**Created**: 2026-06-19
**Status**: Draft
**Input**: User description: "使用 /ui-ux-pro-max 优化后台系统UI"

## Clarifications

### Session 2026-06-19

- Q: 多标签页同时编辑同一表单的冲突处理策略？ → A: 采用 last-write-wins（最终保存胜出），不引入跨标签页冲突检测。理由：后台为少量内部管理员使用，跨标签页编辑同一配置的概率低，复杂同步机制成本高于收益；保留单页内的"未保存确认"（FR-020）即可防止误丢失。
- Q: 仪表盘指标卡片"同期对比"的基准？ → A: 环比昨日（今日 vs 昨日），展示数值与百分比变化。理由：实现成本最低，对"今天是否需要干预"的判断最直接；趋势性判断由已有的趋势图承担，卡片上的对比保持轻量。
- Q: 侧边栏分组的展开/折叠记忆策略？ → A: 自动展开当前页所在分组 + 持久化记忆用户手动调整的其他分组状态（localStorage）。理由：兼顾"自动定位"与高频管理员的个性化偏好，避免每次切换都重置；实现成本可控。
- Q: 全局快速跳转（FR-007）的实现深度？ → A: 采用 Cmd/Ctrl+K 快捷键唤起的命令面板（含可发现性提示）。理由：现代后台/编辑器事实标准（GitHub、Linear、Notion、VS Code），不占顶部空间，对键盘用户友好；FR-007 由 SHOULD 升级为 MUST。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理员快速掌握系统运行状态 (Priority: P1)

作为后台管理员，我每天打开后台第一眼就能清晰看到系统核心运行指标（资源、浏览、搜索、异常、待办），并能在 5 秒内判断"今天是否需要干预"，而不需要在多个折叠菜单和分散卡片之间来回扫视。

**Why this priority**: 仪表盘是管理员的"指挥中心"，是日常使用频率最高的入口。当前仪表盘信息密度低、视觉层次混乱（强渐变抢镜、卡片样式三套并存），导致管理员无法快速建立系统运行的全局心智模型，直接影响日常运维效率。这是本次优化的最高价值切片，独立交付即可显著提升管理员的日常体验。

**Independent Test**: 完成后，管理员登录后台首页即可在首屏内获取"今日是否需要干预"的明确答案，无需点击或滚动即可完成状态判断，并通过点击任一指标卡片直达对应详情页。

**Acceptance Scenarios**:

1. **Given** 管理员登录后台首页，**When** 页面加载完成，**Then** 首屏内一次性展示资源、浏览、搜索三类核心指标的当日数值与同期对比，以及"待处理事项"（如待处理资源、待审核举报、失败任务）的醒目提示
2. **Given** 仪表盘已展示，**When** 管理员点击任一指标卡片，**Then** 平滑跳转到对应的详情页（如资源管理、搜索统计）
3. **Given** 系统存在异常或待办（如失败任务、待审核举报），**When** 管理员查看仪表盘，**Then** 异常/待办以统一且醒目的视觉样式聚合展示，并可一键跳转处理
4. **Given** 首页处于暗色模式，**When** 管理员切换主题，**Then** 仪表盘所有卡片、图表、文字、图标颜色全部正确适配，无对比度不足或元素丢失

---

### User Story 2 - 管理员高效浏览与查找后台功能 (Priority: P2)

作为后台管理员，我能在任何页面快速知道"我在哪里"、"我能去哪里"，并通过清晰的导航、面包屑、全局搜索等方式在 3 次点击内到达目标功能页，而无需记忆菜单层级或反复展开折叠分组。

**Why this priority**: 后台包含 30+ 个功能页面，分属仪表盘、数据管理、系统配置、运营管理、统计分析 5 个分组。当前侧边栏分组默认全折叠、无面包屑、无快速跳转，新管理员难以发现功能入口，老管理员也依赖记忆。导航体验是支撑所有其他功能被有效使用的基础设施。

**Independent Test**: 完成后，从任一页面出发，管理员可通过面包屑/侧边栏/全局搜索在 3 次点击内到达任意目标功能页，且始终明确当前所在位置。

**Acceptance Scenarios**:

1. **Given** 管理员位于任意二级页面（如 `/admin/resources`），**When** 查看页面顶部，**Then** 看到清晰的面包屑路径（如 "管理后台 / 数据管理 / 资源管理"），且每级可点击返回
2. **Given** 管理员首次进入后台，**When** 查看侧边栏，**Then** 当前页面所属分组自动展开并高亮，其他分组保持可识别的折叠态，分组标题清晰可读
3. **Given** 管理员需要跳转到某功能但记不清在哪个分组，**When** 在全局搜索/快速跳转中输入功能名称（如"版权"），**Then** 系统返回匹配的功能入口并可直达
4. **Given** 管理员在窄屏设备上使用后台，**When** 窗口宽度变化，**Then** 侧边栏自动收起为图标态或抽屉态，主内容区自适应，关键操作始终可达

---

### User Story 3 - 管理员在列表页高效处理批量任务 (Priority: P2)

作为后台管理员，我在资源管理、举报管理、待处理资源等列表页能清晰看到筛选条件、当前结果数、已选项数，并通过统一的批量操作入口（如批量删除、批量转移、批量推送）完成任务，而无需在每个列表页重新学习交互方式。

**Why this priority**: 列表页是管理员日常最高频的工作场景（资源管理、待处理、举报、版权、账号、文件等十余个列表页）。当前各列表页筛选栏样式、批量操作入口、空状态、加载态、选中态反馈各写各的，管理员每切换一个列表页都要重新适应，效率损耗显著。统一列表页交互是提升整体后台效率的关键杠杆。

**Independent Test**: 完成后，管理员在任一列表页都能以相同的视觉与交互模式完成"筛选 → 选中 → 批量操作 → 反馈"的全流程，无需重新学习。

**Acceptance Scenarios**:

1. **Given** 管理员进入任一列表页（资源/举报/待处理等），**When** 查看页面，**Then** 看到统一的筛选栏样式（搜索框 + 筛选下拉 + 操作按钮布局一致）、统一的"共 N 项，已选 M 项"提示、统一的批量操作入口位置
2. **Given** 管理员选中若干条目，**When** 触发批量操作（如批量删除），**Then** 出现统一的二次确认交互，操作完成后出现统一格式的成功/失败反馈
3. **Given** 列表查询中，**When** 数据加载，**Then** 显示统一的加载态样式；**When** 结果为空，**Then** 显示统一的空状态（含图标、说明文字、可选的"去添加"引导）
4. **Given** 管理员在列表页进行翻页或调整每页条数，**When** 操作后，**Then** 已选中的条目状态被合理保留或明确提示，不会因翻页静默丢失选中

---

### User Story 4 - 管理员在配置类页面快速完成表单填写 (Priority: P3)

作为后台管理员，我在站点配置、功能配置、开发配置、SEO 配置等表单页能快速理解每个字段的作用，得到即时校验反馈，并明确知道"保存成功 / 保存失败 / 哪个字段有问题"，而无需在冗长的表单中盲目滚动或提交后才报错。

**Why this priority**: 配置类页面影响系统能力开关与站点行为，但当前表单字段密集、缺少分组与说明、校验反馈滞后，管理员填写易出错且出错后定位困难。优先级低于前三者是因为配置类页面使用频率相对较低，但优化后能显著降低误配置风险。

**Independent Test**: 完成后，管理员在任一配置页填写表单时，能在字段失焦或修改时获得即时校验提示，并在保存后获得明确的成功/失败反馈与错误定位。

**Acceptance Scenarios**:

1. **Given** 管理员在任一配置页，**When** 查看表单，**Then** 字段按语义分组（如"基础信息"、"高级选项"），每组有简短说明，必填项有统一标识
2. **Given** 管理员填写了不合法的值（如格式错误、超范围），**When** 字段失焦或修改，**Then** 即时显示校验错误，错误位置与原因明确，无需提交后才得知
3. **Given** 管理员点击保存，**When** 操作完成，**Then** 出现统一的成功/失败反馈；失败时明确指向出错字段
4. **Given** 管理员修改了未保存的内容并尝试离开页面，**When** 触发离开，**Then** 系统弹出统一的"未保存确认"提示，防止误丢失

---

### Edge Cases

- 当后台数据接口返回空（如新建系统首次进入），仪表盘与列表页是否能优雅降级为引导态而非"0 / 0"或空白？
- 当数据加载失败（网络异常、鉴权失效），用户看到的是友好的错误态与重试入口，还是空白或控制台报错？
- 当管理员在窄屏（手机/平板）访问后台，关键操作（导航、筛选、批量操作、保存）是否仍然可用且不出现错位？
- 当管理员开启系统级暗色模式时，所有页面（含图表、状态徽标、富文本编辑器嵌入区域）是否完整适配，无白底/黑字等对比度问题？
- 当管理员在多个浏览器标签页同时编辑同一配置（如一个标签页修改了配置，另一个标签页还显示旧值并保存），系统采用 **最终保存胜出（last-write-wins）** 语义：后保存者覆盖先保存者，不主动检测跨标签页冲突（已在 Clarifications 中确认为可接受范围）。需在用户文档/帮助中提示此行为，避免意外丢失
- 当管理员使用键盘全程操作后台（Tab/Enter/Esc），所有主要交互（菜单、模态框、批量操作、表单）是否可访问？
- 当后台某页面的趋势图数据全为 0（如系统刚部署），图表是否显示"暂无数据"的友好提示而非空坐标轴？

## Requirements *(mandatory)*

### Functional Requirements

#### 视觉一致性

- **FR-001**: 系统 MUST 在所有后台页面建立统一的视觉语言，包括统一的色彩体系（主色、状态色、中性色）、统一的间距与圆角规范、统一的卡片与容器样式，消除"同一页面多种卡片样式并存"的现象
- **FR-002**: 系统 MUST 统一图标库的使用方式（图标集、尺寸、颜色、对齐），消除图标风格混用（如尺寸不一、对齐错位）
- **FR-003**: 系统 MUST 确保首页欢迎区/统计区/图表区/平台区的视觉权重与系统主色协调，消除当前首页"蓝紫渐变抢镜、与蓝色主色冲突"的问题
- **FR-004**: 系统 MUST 在所有页面完整适配明/暗两种主题模式，杜绝部分元素未适配导致的对比度问题或不可读区域

#### 信息架构与导航

- **FR-005**: 系统 MUST 在所有后台页面（除首页外）提供面包屑导航，清晰展示"管理后台 > 分组 > 当前页"的层级，且每级可点击返回
- **FR-006**: 系统 MUST 让侧边栏在用户进入某分组页面时自动展开该分组并高亮当前项；同时 MUST 持久化记忆用户手动调整的其他分组展开/折叠状态（跨会话保留），避免每次切换都重置
- **FR-007**: 系统 MUST 提供 Cmd/Ctrl+K 快捷键唤起的全局命令面板，允许管理员通过输入功能名称（支持中英文模糊匹配）直达目标页面；面板需包含可发现性提示（如首次进入时的引导或顶部状态栏的"按 ⌘K 搜索"提示），避免快捷键无人知晓
- **FR-008**: 系统 MUST 在窄屏场景下将侧边栏收起为图标态或抽屉态，并保证主内容区与关键操作始终可达

#### 仪表盘信息密度

- **FR-009**: 系统 MUST 在首页首屏内聚合展示核心运行指标（资源、浏览、搜索的当日数值与**环比昨日**的对比，含数值差与百分比变化）与待办/异常聚合区（待处理资源、失败任务、待审核举报等），使管理员无需翻页即可判断"今日是否需要干预"
- **FR-010**: 系统 MUST 让仪表盘的每个指标卡片可点击直达对应的详情页
- **FR-011**: 系统 MUST 在仪表盘趋势图为空（数据全 0 或加载失败）时显示友好的"暂无数据/加载失败"状态，而非空坐标轴或硬编码模拟数据
- **FR-012**: 系统 MUST 移除生产代码中残留的调试输出（如控制台日志）与硬编码的模拟 fallback 数据

#### 列表页交互统一

- **FR-013**: 系统 MUST 在所有列表页（资源管理、待处理资源、举报管理、版权申述、平台账号、文件管理、失败资源、未转存资源等）采用统一的筛选栏布局、统一的"共 N 项 / 已选 M 项"提示、统一的批量操作入口位置
- **FR-014**: 系统 MUST 为所有列表页提供统一的加载态、空状态、错误态样式与文案风格
- **FR-015**: 系统 MUST 在批量操作（删除、转移、推送等）前提供统一的二次确认，并在操作完成后提供统一格式的成功/失败反馈
- **FR-016**: 系统 MUST 在列表翻页/调整每页条数时合理保留或明确提示已选项，避免静默丢失选中导致误操作

#### 表单与反馈

- **FR-017**: 系统 MUST 在配置类页面（站点配置、功能配置、开发配置、SEO 配置等）将字段按语义分组并提供简短说明，必填项使用统一标识
- **FR-018**: 系统 MUST 在表单字段失焦或修改时提供即时校验反馈（而非仅在提交后报错），并明确指示出错字段与原因
- **FR-019**: 系统 MUST 在所有保存/提交操作后提供统一格式的成功/失败反馈
- **FR-020**: 系统 MUST 在用户修改未保存内容并尝试离开页面时弹出统一的"未保存确认"提示

#### 可访问性与响应式

- **FR-021**: 系统 MUST 让所有主要交互（菜单、模态框、批量操作、表单、导航）支持键盘操作（Tab/Enter/Esc）
- **FR-022**: 系统 SHOULD 为纯图标按钮提供可访问的文本说明（如 aria-label），确保屏幕阅读器可识别
- **FR-023**: 系统 MUST 保证文本与背景的色彩对比度满足可访问性基线（正文/重要信息在两种主题下均可清晰阅读）

### Key Entities *(include if feature involves data)*

- **后台主题模式**: 明色 / 暗色 两种主题，所有页面元素需同时适配；切换由用户偏好或系统设置驱动
- **导航分组**: 仪表盘、数据管理、系统配置、运营管理、统计分析 共 5 个分组，每组包含若干功能页；分组支持折叠/展开与当前页高亮；用户的展开/折叠偏好需持久化（跨会话保留），进入某分组页时该分组自动展开
- **指标卡片**: 仪表盘的核心展示单元，包含当日数值、标签、**环比昨日对比（数值差 + 百分比变化，含上升/下降方向标识）**、跳转目标；可点击
- **列表页通用模式**: 抽象出"筛选栏 + 结果统计 + 列表内容 + 批量操作"的统一模式，适用于所有列表型页面
- **表单分组**: 配置类页面的字段组织单元，每组有标题与说明，支持折叠/展开

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员登录后台首页后，无需滚动或点击，在首屏 5 秒内即可获取"今日是否需要干预"的明确判断（依据：核心指标 + 待办/异常聚合区一次性可见）
- **SC-002**: 管理员从后台任一页面出发，到达任意目标功能页的平均点击次数 ≤ 3 次（依据：面包屑 + 自动展开的侧边栏 + 全局快速跳转）
- **SC-003**: 后台所有页面（30+ 个）在视觉一致性检查中通过率 ≥ 95%，具体表现为：同一类元素（卡片、按钮、筛选栏、空状态、加载态、反馈提示）在所有页面采用同一套样式规范
- **SC-004**: 明/暗两种主题模式下，所有后台页面的文本对比度均达到可访问性基线（正文对比度 ≥ 4.5:1），无不可读区域或对比度不足的元素
- **SC-005**: 管理员在列表页完成"筛选 → 选中 → 批量操作 → 反馈"全流程的学习成本降低，新管理员首次接触任意列表页即可正确完成批量操作（无需查阅文档），通过率 ≥ 90%
- **SC-006**: 配置类页面的表单错误率（提交后才发现的校验失败）相比现状下降 ≥ 50%，依据：字段即时校验与分组说明降低了误填概率
- **SC-007**: 生产环境后台代码中不再残留调试输出（控制台日志）与硬编码的模拟 fallback 数据
- **SC-008**: 后台在窄屏（≤ 768px 宽度）下，关键操作（导航开关、筛选、批量操作、保存）100% 可达且不错位

## Assumptions

- 后台的目标用户为系统管理员（少量、内部、熟练计算机操作），不面向终端访客；优化重点在于"专业、高效、低干扰"，而非"营销吸引力"
- 后台现有功能范围（30+ 个页面、5 个导航分组）在本次优化期间保持稳定，不涉及大规模功能增删；本规范聚焦"现有功能的 UI/UX 优化"，不新增业务功能
- 后台已具备账号密码鉴权机制（admin / password1），本规范不涉及鉴权流程改造，仅在视觉层面确保登录态/退出态反馈一致
- 现有后端 API 与数据结构保持不变；本规范不修改后端接口，仅在必要时调整前端如何展示与消费现有数据
