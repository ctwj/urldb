# Feature Specification: 转存文件定时自动清理

**Feature Branch**: `002-auto-cleanup-transfer`
**Created**: 2026-06-14
**Status**: Draft
**Input**: User description: "当前系统自动转存功能，只支持 quark，只进行了转存和分享， 是否能够支持 一定时间后， 自动删除转存的文件"

## Clarifications

### Session 2026-06-14

- Q: 资源的清理状态应该用哪种数据模型表达（布尔/枚举/历史记录表）？ → A: 不新增独立的清理状态字段；复用现有资源上由转存写入的字段（fid、转存链接等），清理成功后清空这些字段即代表"已清理"，资源是否已清理通过这些字段是否为空来判定。
- Q: 清理失败后的重试次数上限与超限处理？ → A: 不实现专门的重试机制。清理失败时仅记录失败原因（与失败时间）；下一轮定时任务运行时，对"fid 仍存在且转存时间仍超期"的资源会自然再次尝试删除——这是定时扫描的固有行为，不属于"重试"。无重试次数上限、无"待重试"状态。
- Q: 保留时长与调度周期的默认值？ → A: 两者均可由管理员配置；保留时长默认 7 天，调度周期默认 1 小时。
- Q: 是否支持手动触发清理任务？ → A: 不支持。系统不提供手动触发入口，完全依赖定时调度；管理员如需立即清理需等待下一个调度周期或调整调度周期配置。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 按保留时长自动清理转存文件 (Priority: P1)

作为系统管理员，我希望系统能够在转存文件保存达到设定的保留时长后，自动从网盘账号中删除这些文件，从而避免账号存储空间被长期占用、降低被风控的风险，而无需我手动逐个清理。

**Why this priority**: 这是本功能的核心价值——释放网盘账号存储空间、避免长期占用导致的账号风险。没有这一项，整个功能就没有存在的意义。

**Independent Test**: 在系统中配置一个较短的保留时长（例如 1 天）与较短的调度周期，等待一条已完成转存的资源超过保留时长并经过一轮调度周期，验证网盘账号中对应的文件被删除，且资源上由转存写入的字段（fid、转存链接）被清空。

**Acceptance Scenarios**:

1. **Given** 一条资源的转存时间距今已超过系统配置的保留时长（如 7 天），**When** 清理任务运行，**Then** 该资源对应的网盘文件从账号中被删除，资源上由转存写入的字段（如 fid、转存链接）被清空，并记录清理时间。
2. **Given** 一条资源的转存时间距今未超过保留时长，**When** 清理任务运行，**Then** 该资源不被处理，保持原状。
3. **Given** 系统管理员未启用自动清理功能，**When** 清理任务运行，**Then** 任务立即退出，不删除任何文件。
4. **Given** 清理任务删除某文件时网盘返回失败（如 cookie 失效、网络异常），**When** 任务结束，**Then** 失败原因被记录在该资源上，任务继续处理其他资源不被阻塞；下一轮定时任务运行时该资源（fid 仍存在且仍超期）会被自然再次纳入清理尝试。

---

### User Story 2 - 灵活配置保留时长与启用开关 (Priority: P2)

作为系统管理员，我希望可以在管理后台开启/关闭自动清理功能，并灵活设置转存文件的保留时长（例如 3 天、7 天、30 天），以便根据我的网盘账号容量和使用场景调整清理策略。

**Why this priority**: 没有可配置项，P1 的清理逻辑就无法适配不同场景；但清理逻辑本身（P1）是先决条件，配置能力紧随其后。

**Independent Test**: 在管理后台修改保留时长与启用开关，验证下一次清理任务运行时严格按新配置执行（启用时清理、关闭时不清理；时长边界判定正确）。

**Acceptance Scenarios**:

1. **Given** 管理员在后台将保留时长从 7 天改为 3 天，**When** 配置保存成功后下一次清理任务运行，**Then** 转存时间超过 3 天的资源被纳入清理范围。
2. **Given** 管理员关闭了自动清理开关，**When** 清理任务被调度，**Then** 任务立即返回，不执行任何删除。
3. **Given** 管理员设置了一个非法的保留时长（如 0 或负数），**When** 保存配置，**Then** 系统拒绝保存并给出明确的错误提示。

---

### User Story 3 - 清理过程可观测、可追溯 (Priority: P3)

作为系统管理员，我希望能够查看每次清理任务的执行情况，包括本次清理了哪些资源、哪些清理失败及失败原因，以便我了解系统运行状态并对失败案例进行人工干预。

**Why this priority**: 可观测性是运维必要能力，但属于清理功能稳定运行之后的增强项，因此排在 P3。

**Independent Test**: 触发一次清理任务后，查看任务执行日志或资源列表中"已清理"标记与失败原因字段，验证信息准确且可查询。

**Acceptance Scenarios**:

1. **Given** 清理任务已执行完毕，**When** 管理员查看资源列表，**Then** 已清理的资源展示"已清理"状态及清理时间，清理失败的资源展示失败原因。
2. **Given** 清理任务执行过程中发生异常，**When** 管理员查看运行日志，**Then** 可以看到任务开始时间、处理数量、成功数量、失败数量及任务结束时间。

---

### Edge Cases

- 当资源转存到网盘后、清理任务运行前，文件已被用户在网盘端手动删除，系统应识别"文件不存在"并视为清理成功，避免反复重试。
- 当网盘账号 cookie 失效导致删除接口返回鉴权错误时，该资源应记录失败原因并跳过，不应阻塞队列中其他资源的清理；下一轮任务运行时若账号 cookie 已恢复则自然再次尝试。
- 当系统时钟出现回拨或资源转存时间字段缺失时，应跳过该资源并记录告警，而非抛出异常中断整个清理任务。
- 当同一资源的清理任务被并发触发时（例如手动触发与定时触发重叠），必须有互斥机制，避免对同一文件发起重复删除。
- 当管理员在清理过程中修改了保留时长，已进入本轮处理的资源按本轮开始时读取的配置执行，下一轮按新配置执行。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个全局开关，用于启用或禁用"转存文件定时自动清理"功能。
- **FR-002**: 系统 MUST 提供一个可配置的"保留时长"参数（单位：天），用于决定转存文件在网盘账号中保留多久后自动删除；默认值为 7 天。
- **FR-003**: 系统 MUST 在每轮清理任务中，筛选出"转存完成时间 + 保留时长 ≤ 当前时间"且尚未被清理的资源，逐条执行删除。
- **FR-004**: 系统 MUST 调用对应网盘服务的删除能力，从转存目标账号中删除文件本体（释放账号存储空间）。
- **FR-005**: 系统 MUST 在删除成功后清空资源上由转存写入的字段（如 fid、转存链接等），并记录清理完成时间；"已清理"状态通过这些转存字段是否为空来判定，不新增独立的清理状态字段。资源数据库条目与原分享链接信息予以保留，不进行物理删除。
- **FR-006**: 系统 MUST 在删除失败时记录失败原因（与失败时间），失败资源不得阻塞队列中其他资源的处理。系统不实现专门的重试机制——每轮定时任务对"fid 仍存在且转存时间超期"的资源尝试删除一次，失败仅记录原因，不进行重试计数或重试队列。
- **FR-007**: 系统 MUST 通过定时调度器周期性触发清理任务，调度周期可由管理员配置（默认 1 小时），且调度周期应可观测（运行日志中体现任务起止与处理统计）。
- **FR-008**: 系统 MUST 提供互斥机制，防止清理任务对同一资源或同一轮次并发重复执行。
- **FR-009**: 系统 MUST 在网盘返回"文件不存在"类响应时，视为清理成功并按成功流程清空转存字段，不再尝试。
- **FR-010**: 管理 UI MUST 提供配置入口，允许管理员开启/关闭自动清理、设置保留时长，并对非法输入（如非正数）给出明确校验提示。
- **FR-011**: 管理 UI MUST 在资源列表中直观区分"已清理"与"未清理"资源，并展示清理时间与失败原因（如适用）。
- **FR-012**: 清理范围 MUST 仅限本系统自动转存产生的文件，不得删除用户手动转存或网盘账号中原本存在的其他文件。

### Key Entities *(include if feature involves data)*

- **资源（Resource）**: 已转存到网盘账号的文件条目。转存时会写入 fid、转存链接等字段；清理时通过清空这些字段表达"已清理"状态，无需新增独立清理状态字段。需扩展记录"转存完成时间""清理时间""清理失败原因""最近一次失败时间"等属性，以支撑清理判定与可观测性（不记录重试次数，无重试机制）。
- **系统配置（System Config）**: 需新增"自动清理开关""保留时长（天，默认 7）""清理任务调度周期（默认 1 小时）"等配置项，用于驱动清理行为。
- **清理任务执行记录（可选）**: 记录每次清理任务的起止时间、处理资源数、成功数、失败数，用于管理员追溯。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 启用自动清理后，转存时间超过保留时长的文件，在下一轮清理任务运行后的 1 个调度周期内被从网盘账号删除，删除成功率达到 95% 以上（排除账号 cookie 失效等外部因素）。
- **SC-002**: 管理员可在 1 分钟内完成"启用清理 + 设置保留时长"的全部配置，配置立即生效，无需重启服务。
- **SC-003**: 清理任务运行后，转存目标网盘账号的已用存储空间出现可观测的下降（在文件量足够时），账号空间占用率较启用前降低。
- **SC-004**: 清理任务的执行不影响系统其他功能（转存、分享、推送、查询）的正常运行，不出现资源列表错乱或分享链接异常失效。
- **SC-005**: 清理失败的资源其失败原因对管理员完全可见；对于因瞬时故障（如网络抖动、临时鉴权失败）导致的失败，下一轮定时任务运行时能够自然再次尝试并最终成功清理（成功率 95% 以上），无需管理员干预。

## Assumptions

- 本期自动清理仅针对当前已支持的网盘平台（Quark / 夸克）转存产生的文件；其他网盘（阿里云盘、百度网盘、UC 网盘等）的转存与清理在各自能力就绪后再纳入范围，本期不阻断其后续接入。
- 保留时长采用"全局统一配置"作为 MVP 方案；按账号/按资源差异化保留时长的能力留待后续迭代，本期不实现。
- "删除文件"指删除转存到目标网盘账号中的文件本体，以释放账号存储空间；数据库中的资源记录、原分享链接信息予以保留，清理后通过清空转存字段（fid、转存链接等）表达"已清理"，不进行物理删除资源条目。
- 默认调度周期为 1 小时（清理属低时效需求，远低于 ready_resource 调度器的分钟级频率，避免对网盘 API 造成额外压力与风控风险）；具体周期值可由管理员配置。
- 系统不提供手动触发清理任务的入口，完全依赖定时调度。
- 网盘服务的删除能力在现有网盘服务接口基础上扩展实现，复用现有 cookie 鉴权与账号管理机制。
- 资源已具备"转存完成时间"或可通过既有字段推算；若现有字段不足，将在规划阶段确认字段补充方案，不影响本规范成立。
- 清理失败不实现专门的重试机制，仅记录失败原因；依赖下一轮定时任务对"fid 仍存在且超期"的资源自然再次尝试。
